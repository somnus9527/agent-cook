# 06 · Memory 与分级记忆架构

> 阶段 3 重点。回答你的疑问：为什么这么多 Agent 在 Memory 上下大功夫、"三级/四级 memory"到底在做什么、为什么要做这么多优化。

## 1. 先理解"为什么 Memory 是个大问题"

回到知识点 02 的事实：**LLM 是无状态的，且上下文窗口有限**。一切 Memory 优化都源于下面这组矛盾：

| 约束 | 带来的问题 |
|---|---|
| 上下文窗口有限（哪怕 128k、1M） | 长对话/长任务装不下全部历史 |
| 上下文越长越**贵**、越**慢** | 不能无脑把所有东西塞进去 |
| 注意力会被稀释 | 塞太多反而"找不到重点"，回答变差（lost-in-the-middle） |
| 模型无状态、会话会断 | 跨会话/重启后"失忆"，没法当长期协作者 |
| 信息有**时效**和**相关性** | 旧的、无关的信息要能被更新/遗忘/按需召回 |

**所以 Memory 的本质任务是**：在"有限且昂贵的上下文窗口"里，**只放当前最该放的东西**，其余的存到外部、需要时再精准取回。这正是操作系统管理"有限 RAM + 海量磁盘"的同一个问题——这也是为什么后面的架构大量借用 OS 隐喻。

## 2. 记忆的分类（认知科学类比）

业界常按人脑记忆来组织 Agent 记忆：

- **工作记忆 / 短期记忆**：当前在上下文窗口里的内容（当前对话、scratchpad）。
- **长期记忆**，又分：
  - **情景记忆 (episodic)**：发生过的事/过往交互（"上次你让我用 TS"）。
  - **语义记忆 (semantic)**：事实与知识（"用户偏好简洁回答"、领域知识）。
  - **程序性记忆 (procedural)**：技能/怎么做（可复用的"skill"、工具用法）。

（这套划分来自 CoALA 等"语言智能体认知架构"工作。）

## 3. 分级记忆架构：源头是 MemGPT

**MemGPT**（2023，*LLMs as Operating Systems*；现已演进为开源项目 **Letta**）是这条线的开山之作。核心思想：**把上下文窗口当成被管理的内存资源**，照搬 OS 的 RAM/Disk 分层：

```
主上下文 Main Context（≈ RAM，模型能直接"看到"的固定大小 prompt）
  ├─ 系统指令（固定：规则 + 工具 schema）
  ├─ 工作上下文（scratchpad：推理中间结果）
  └─ FIFO 消息缓冲（最近若干轮对话）

外部上下文 External Context（≈ Disk，几乎无限，模型默认看不到，要显式检索）
```

**最关键的创新**：让 **LLM 自己管理分页**。当主上下文快满（如 70%）时，系统给模型发一个内部告警，模型**自己决定**哪些内容不重要 → 摘要后写入外部存储；需要旧信息时，**自己调用工具**把它检索回主上下文。即"模型用 function call 给自己换页"，是一种原始的认知自我调节。

## 4. "三级 / 四级 memory" 具体指什么

你听到的多级架构，基本都是 MemGPT 的后继者把"分层"做得更明确：

### 三级 → MemoryOS
更刚性的三层存储：
1. **STM 短期记忆**（Short-Term）：实时对话。
2. **MTM 中期记忆**（Mid-Term）：按话题聚合的摘要。
3. **LPM 长期个人记忆**（Long-term Personal Memory）：持久化的用户画像 / Agent 人设。

信息随时间和重要性，从 STM → MTM → LPM **逐级沉淀/固化**（类似 CPU 缓存 L1/L2/L3 或人脑的记忆巩固）。

### 四级 → H-MEM
按**语义抽象度**分四层，结构类似文档的 章 / 节 / 小节 / 正文：
- 高层是高度概括的主题索引，逐层向下越来越具体，最底层是原始内容。
- 好处：检索时先命中高层主题，再逐层下钻，**提高检索精度与效率**（解决 MemGPT 外部记忆"纯向量检索精度/效率有取舍"的问题）。

> 所以"几级"不是越多越好，而是不同团队在"如何组织外部记忆、如何在精度/效率/成本间权衡"上给出的不同切法。

## 5. 为什么要做这么多优化（各派在优化什么）

不同系统针对 MemGPT 暴露的不同短板下功夫——这正是"花大力气"的原因：

| 系统 | 主要优化点 |
|---|---|
| **MemGPT/Letta** | 提出 OS 分层 + 自主分页（奠基） |
| **MemoryOS** | 把分层显式化为 STM/MTM/LPM 三级，明确巩固路径 |
| **H-MEM** | 按语义抽象度分四层，提升**检索精度与效率** |
| **MemTier** | 巩固改为**异步守护进程**（非中断触发）+ **RL** 自适应检索策略 |
| **Mem0** | **动态巩固**记忆状态；在 Agent 记忆基准上名列前茅；可作很多系统的记忆后端 |
| **Zep** | 基于**时序知识图谱**，擅长处理"事实随时间变化" |
| **MAPLE** | 指出关键缺口：MemGPT 只解决"存储"，没解决"从记忆里提炼智能/个性化" |

归纳这些优化的几个**维度**（也是你评估任何 Memory 方案的清单）：
1. **存什么 / 怎么提炼**：原文？摘要？抽取成结构化事实/知识图谱？
2. **怎么组织**：扁平向量库 vs 分级 vs 图。
3. **何时巩固/遗忘**：同步中断 vs 异步后台；如何更新过时信息、淘汰无关信息。
4. **怎么召回**：相似度检索、按 recency/importance/relevance 打分、分层下钻。
5. **谁来决策**：模型自管（MemGPT）vs 外部规则/独立服务。

## 6. Hermes Agent（你提到的"最有名的"那个）

**Hermes Agent**（Nous Research 出品）是一个开源、自我改进的 Agent，定位是"长期协作者"而非健忘的聊天机器人。它的记忆设计很有代表性：

- **持久化文件**（存于 `~/.hermes`，跨会话/重启/更新存活）：
  - `SOUL.md`：人设/persona。
  - `MEMORY.md`：Agent **自己**curated 的事实笔记——它根据对话**自主**决定记什么（你的项目、偏好、环境、工作流），不靠你显式下命令。
  - `state.db`：会话数据库。
- **系统提示构造**：开新对话时，由核心指令（工具定义、行为规则，约 10K token，固定）+ 记忆文件拼出。
- **反思 pass（reflection）**：周期性地"回看"——读最近会话，提炼模式、写回 `MEMORY.md` 或生成新的可复用 skill。效果：忙了一天后，`MEMORY.md` 会多出一条总结昨天要点的条目，相当于 Agent "睡了一觉想清楚了什么重要"。
- **外部记忆 provider 生态**：Hermes 支持接 8 种外部记忆后端，包括 **Mem0**（服务端用 LLM 抽取记忆）和 **Honcho**（推断**隐式**理解，如"这人偏好简洁回答"，而 MEMORY.md 只存显式写下的）。
- 社区还有 **Memory OS**（7 层、本地跑、Qdrant + Redis）叠在 Hermes 之上，把记忆基础设施本地化，对标云端的 mem0 / Zep / Letta。

> 注意：还有个完全无关的 "Hermes" 是 NousResearch 的**函数调用模型**系列，以及一篇讲 GPU 内存的硬件论文——别混淆。你说的是 **Hermes Agent** 这个记忆系统。

## 7. 给你的入门建议

别一上来就实现 MemoryOS/H-MEM。按这个梯度：
1. **最小**：就是知识点 02 里那个 `messages` 数组（纯短期记忆）。
2. **加截断/摘要**：上下文快满时摘要旧消息（见知识点 08）——你就实现了 MemGPT 思想的最简版。
3. **加长期记忆**：把"事实/偏好"抽出来存到一个文件或向量库（像 Hermes 的 MEMORY.md），下次开场注入——你就有了 episodic/semantic 记忆雏形。
4. **加检索**：长期记忆量大时，用 RAG（知识点 07）按需召回，而不是全量塞入。

理解这条"为什么 → 分层 → 各派优化"的脉络，比记住某个具体系统更重要。
