# 14 · Embedding 与 RAG 讲透

> 知识点 07 讲 RAG 怎么**实现**；本篇讲它到底**是什么、为什么**，重点把 Embedding 在干嘛讲清楚（只讲原理，不要求实现）。看完再回头看 07 的代码会很顺。

## 一、先理解 Embedding：把"意思"变成"坐标"

### 1. 一句话
**Embedding（嵌入）= 把一段文本变成一串数字（向量），让"意思相近"的文本，数字也相近。**

### 2. 为什么需要它
计算机不懂文字的"意思"，只会算数字。`"猫"` 和 `"小猫咪"` 对字符串匹配来说毫无关系（没有共同字符规则上也对不齐），但它们意思几乎一样。我们需要一种方式让机器能**按意思**比较文本——Embedding 就是干这个的：它把文本投射到一个"意义空间"里的一个点。

### 3. 直观例子：假想一个二维"意义空间"
真实 Embedding 有几百上千维、且维度不可解释，但我们先用一个**人为编的 2 维**例子建立直觉。假设两个轴是：
- 横轴 = "皇室程度"
- 纵轴 = "性别（男←→女）"

那么几个词的"坐标"大概会是（数字是编的，仅为示意）：

```
            性别(女) ↑
                     │      queen(0.9, 0.8)
        woman(0.1,0.9)│
                     │
─────────────────────┼───────────────────→ 皇室程度(高)
                     │
         man(0.1,0.1)│      king(0.9, 0.2)
            性别(男) ↓
```

你会发现一个神奇的事：**`king - man + woman ≈ queen`**（向量加减在意义上成立）。这说明向量真的把"语义关系"编码进了几何位置——这正是 Embedding 的精髓。

> 真实模型里维度是 384 / 768 / 1536… 这种，每个维度不对应"皇室/性别"这种人能命名的概念，而是模型自己学出来的抽象特征。但"**意思近 → 位置近**"这个性质完全一样。

### 4. 怎么比较两个向量"像不像"：余弦相似度
看两个向量的**夹角**：方向越一致越相似。
- 夹角小（余弦≈1）→ 意思很近（`猫` vs `小猫咪`）
- 夹角大（余弦≈0）→ 没什么关系（`猫` vs `财务报表`）

知识点 07 里那个 `cosine(a,b)` 函数算的就是这个。**你不用自己实现 Embedding**——调模型（如 Ollama 的 `nomic-embed-text`）给文本，它返回向量；你只需要会算相似度、做排序。

### 5. Embedding 在干嘛（收束成一句）
**把"按意思检索"这个模糊问题，变成"在向量空间里找最近的点"这个能算的数学问题。** 这是 RAG 能成立的地基。

## 二、RAG：让模型"开卷考试"

### 1. 一句话
**RAG（Retrieval-Augmented Generation，检索增强生成）= 回答前先去知识库里检索相关资料，把资料塞进 prompt，让模型基于资料作答。**

### 2. 为什么需要（模型自己的三个短板）
- **不知道你的私有数据**：公司 PRD、内部规范，模型训练时没见过。
- **知识会过期**：训练有截止时间，之后的事不知道。
- **会一本正经胡编（幻觉）**：不知道也照答。

RAG 的思路就像**开卷考试**：不要求模型"背下所有知识"，而是"考试时把相关那几页书翻给它看，让它照着答"。

### 3. RAG 的完整流程
```
【离线建库】（知识点 12 的知识库）
文档 → 切分(chunk) → 每段算 Embedding → 存进向量库 {文本, 向量}

【在线问答】
用户问题 → 算 Embedding → 在向量库里找最相近的 Top-K 段（按余弦相似度）
        → 把这 K 段拼进 prompt（"参考以下资料回答…"）→ 模型据此作答 + 标注引用
```

关键：**检索靠的是 Embedding 的"按意思找"**，所以用户问"怎么请假"，能命中写着"休假申请流程"的文档——尽管字面不一样。

### 4. 一个端到端的"手算"示例（仅示意，不实现）
知识库里有两段（括号里是它们的"意义坐标"，编的）：
- A：「报销需在月底前提交发票」 → 向量 `(0.8, 0.1)`
- B：「年假可在系统里申请」 → 向量 `(0.1, 0.9)`

用户问：「我怎么休假？」→ 算出问题向量 `(0.15, 0.85)`。
- 与 A 的余弦相似度：低（方向差很多）
- 与 B 的余弦相似度：高（方向几乎一致）

→ 检索命中 **B**，于是拼出 prompt：
```
参考资料：年假可在系统里申请。
问题：我怎么休假？
（请只根据参考资料回答）
```
→ 模型答："你可以在系统里申请年假。" 并标注引用了 B。

**整个过程没有让模型去'记住'公司制度，只是临时把对的那段喂给它。** 这就是 RAG。

### 5. 适用场景（什么时候该用 RAG）
✅ 适合：
- 基于**私有/内部知识**问答（PRD、规范、客服知识库、产品文档）。
- 知识**经常更新**（更新知识库即可，不用重训模型）。
- 需要**可溯源/引用**（答案要能指到出处）。
- 知识量**远超上下文窗口**，没法全塞进 prompt。

❌ 不适合 / 要换思路：
- 要的是**执行一个确定动作**（查实时库存、调 API、跑计算）→ 用 **tool**（知识点 03），不是 RAG。
- 知识量很小、能直接塞进 prompt → 直接放进上下文即可，不必建库。
- 需要模型**改变行为风格/格式**（而非补充知识）→ 那是 prompt 工程或微调的活，不是 RAG。

### 6. RAG vs 其它"喂知识"的方式（对比记忆）
| 方式 | 怎么做 | 适合 |
|---|---|---|
| **RAG** | 检索相关片段塞进 prompt | 私有/海量/常更新的知识，要溯源 |
| **超长上下文** | 把全部资料直接塞进 prompt | 资料不大、一次性 |
| **微调 (fine-tune)** | 用数据再训练模型 | 改风格/格式/固定技能，不适合频繁变动的事实 |
| **Tool 调用** | 让模型调函数实时取 | 动态数据、要执行动作 |

> 它们可叠加：很多真实系统 = RAG（取知识）+ Tool（取实时数据/执行）+ 好 prompt。

## 三、RAG 的两个重要局限（很多人栽在这）

### 3.1 RAG 不是幻觉的"根因"，只是"缓解药"
纠正一个常见的因果误解：**不是 RAG 导致幻觉，而是 LLM 本来就会幻觉，RAG 用来缓解它。**

幻觉的**根本原因在 LLM 自己**（呼应知识点 02 的心智锚点）：它是个**概率性的 next-token 预测器**，目标是"生成看起来最可能的下一个词"，而**不是"说真话"**。它没有内建的事实校验，不知道的也会流畅地编——因为"编得通顺"和"是真的"在它的机制里是两回事。

RAG 的作用：把相关资料放进上下文，让模型"照着资料答"，从而**降低**幻觉。但它**治标不治根**，且自身会引入新的失败点：
- 检索回来的资料**本身就错/不相关** → 模型照着错答（垃圾进垃圾出）。
- 检索**没找到**关键信息 → 模型退回"自己编"。
- 资料找对了但模型**没好好用**，仍按先验答 → 要靠 prompt 约束"只依据资料"+ 引用溯源来压。

→ RAG 能显著减少幻觉，但**不能消灭**。要进一步压：限定"无依据就说不知道"、引用可溯源、校验（知识点 10）、评估（知识点 12 §4.8）。

**进一步：把不确定性看成"系统性的"，而非"单点的"。** 常见误解是"LLM 幻觉 + RAG 幻觉两个东西相加"。更准的图景是——**LLM 的概率生成本质是底层地基（天生不确定）**；你往上加的每个组件都在把它约束向正确，但**各自又引入新的失败面**，不确定性沿整条流水线**累积/放大**：

```
概率生成的 LLM（地基不确定）
  + 检索可能取错/取漏（RAG 的失败面）
  + 工具可能报错/返回脏数据（tool 的失败面）
  + 上下文可能被压坏/被注入（memory/context 的失败面）
  = 整个系统的不确定性
```

关键认知：**它压不到零，只能"圈住和管理"。** 校验、引用溯源、评估、人在环、可观测——整个 Agent 工程学科，本质就是在给这份固有的不确定性兜底，把它圈进可接受的边界。这就是为什么不能"只搭个能跑的 demo 就上生产"。

### 3.2 向量检索处理不了"逻辑链条"（多跳推理）
这是朴素 RAG 最关键的盲区。看一个例子——

知识库三条：
- **A**：小明读九年级
- **B**：九年级三月份有期末考试
- **C**：图书馆在期末考试前两周关闭

问题：**"图书馆下周开门吗？"**

要答对，必须**串起一条推理链**：
```
小明 →(A)→ 九年级 →(B)→ 三月有期末考试 →(C)→ 考前两周图书馆关闭 → 推算日期 → 回答
```

但向量检索是怎么做的？它把**问题**变成向量，去找**语义最近**的文本块。问题的关键词是"图书馆/下周/开门"（带"小明"的话还有小明）：
- **C** 含"图书馆""关闭" → 语义很近，**命中** ✅
- **A** 含"小明" → **命中** ✅
- **B** 既没"图书馆"也没"小明"，只讲"九年级""期末考试" → 和问题语义不近，**被漏掉** ❌

而 **B 恰恰是连接 A 和 C 的那座"桥"**。桥断了，模型拿到 A 和 C 也拼不出链条，于是要么说不知道，要么开始幻觉。

**根本原因：向量相似度衡量的是"语义接近"，不是"逻辑关联"。** 推理链里的中间环节，往往在字面/语义上离问题很远，所以被漏检。这类**多跳（multi-hop）**问题，是朴素 RAG 的天然短板。

**怎么破（了解方向即可）：**
- **多跳 / 迭代检索（Agentic RAG）**：不一次检索完，让 Agent **检索→推理→再检索**。先查"小明读几年级"→九年级→再查"九年级有啥考试"→期末考试→再查"期末考试与图书馆"→关闭。把单次相似度检索，变成 Agent 主导的多轮推理（正是知识点 04 的循环 + 把检索做成工具）。
- **GraphRAG / 知识图谱**：把信息存成**实体+关系**的图（小明—属于→九年级—有→期末考试—触发→图书馆关闭），检索时**沿关系"走"**而非只算语义距离，天生适合多跳。
- **查询分解（query decomposition）**：先让 LLM 把复杂问题拆成子问题，分别检索再汇总。

→ 一句话：**朴素向量 RAG 擅长"找相似"，不擅长"串逻辑"。需要逻辑链/多跳时，用 Agentic RAG 或知识图谱。**
