# 12 · 知识库（Knowledge Base）

> 阶段 3。知识点 07（RAG）讲的是"查了再答"这套**动作**；本篇讲被查的那个**东西**——知识库：它是什么、工程上怎么搭、怎么优化。

## 1. 概念：先把三个词分清

| 概念 | 是什么 | 类比 |
|---|---|---|
| **知识库 (Knowledge Base)** | 你整理、结构化、可检索的**数据资产**本身（语料 + 接入管线 + 存储 + 索引） | 一座**图书馆**（书 + 编目 + 书架） |
| **RAG**（知识点 07） | 查询时**从知识库取相关片段、拼进 prompt 让模型作答**的技术流程 | **借阅+查阅**的动作 |
| **Memory**（知识点 06） | 记住与**用户/任务**相关的事实并跨会话复用 | 你自己的**笔记本** |
| 模型内置知识 | 训练时学进参数里的 | 馆员脑子里记得的 |

一句话：**RAG 是读法，知识库是被读的馆藏。** RAG 站在知识库之上；知识库质量差，RAG 再花哨也答不准（garbage in, garbage out）。Memory 在技术上常和知识库共用向量检索，但用途不同（私人、可变 vs 相对稳定的共享知识）。

### 1.1 什么进知识库、什么做成 tool（含一个易错点）

判断标准是"要 LLM **知道**还是要它**做**"：
- 纯文本、靠"读懂"产生价值（PRD、规范、设计文档）→ **知识库 + RAG**。
- 有确定行为、运行时执行（查实时库存、调 API）→ **tool**（知识点 03）。

**易错点：tool 其实有两种含义，别混。** 一个"通用能力"（如内部 npm 包）该怎么放，取决于它是被**运行时调用**还是被**织入生成的代码**：

| | 模式 A：运行时调用 | 模式 B：织入生成的代码 |
|---|---|---|
| 谁在用 | **Agent 自己**（推理中调它拿结果） | **Agent 产出的项目代码**（import 使用） |
| 例子 | "发短信"能力，Agent 真的调它发出去 | 前端组件/工具库，生成的页面里 import |
| 该做成 | **runtime tool** | **结构化能力目录**（供检索后照着生成代码）+ 可选**脚手架 tool** |

- 模式 B 的资产**不是 runtime tool**：Agent 不会"调用"一个 UI 组件库，而是要**知道它存在、它的 API、何时该用**，从而生成正确的 `import` + 用法。
- 它需要的是**结构化目录/索引**（名称、场景标签、API 签名、示例、版本），**不是对 README 做模糊 RAG**——用 `package.json`/类型/示例这些结构化信息。
- 可选再加"脚手架 tool"：`scaffold(组件X)` 吐出样板代码；注意这里的 tool 是"生成脚手架"这个动作，**不是那个包本身**。

→ 收束：**运行时被调用 → tool；纯文本靠读懂 → 知识库；要被生成进代码的构件 → 结构化目录(+脚手架)。**

## 2. 工程组成（一个知识库由这些部件构成）

```
数据源 → 接入管线(Ingestion/ETL) → 存储 → 索引 → 检索接口 →（交给 RAG）
```

1. **数据源**：文档（PDF/Word/Markdown/HTML）、网页、数据库、API、代码库……
2. **接入管线 Ingestion（离线，最花功夫的一环）**：
   `加载(load) → 解析(parse 成纯文本) → 清洗(去页眉页脚/乱码) → 切分(chunk) → 向量化(embedding) → 写入存储`
3. **存储**（通常三者并存）：
   - **向量** + **元数据**（来源、标题、章节、时间、权限标签…）
   - **原文档**（document store，便于回溯整段/整篇）
   - 可选：**关键词倒排索引**（BM25，做混合检索用）、**知识图谱**（结构化关系）
4. **索引 Index**：让"给一个查询向量，快速找最近 K 条"成立（向量库内部用 HNSW 等近似最近邻）。
5. **检索接口 retrieve(query) → 片段**：RAG 调它。

## 3. 工程实现：从最小到生产

### 最小版（先这么做，理解全链路）
就是知识点 07 的内存数组 `[{text, vector, meta}]`：手写 load→chunk→embed→push，检索用余弦相似度排序。**麻雀虽小五脏俱全**，先把管线跑通。

### 生产版（量大、要持久化、要并发时升级）
- **向量库换专用存储**：sqlite-vec / LanceDB（本地轻量）、pgvector（已有 PG）、Qdrant / Milvus / Chroma（专用）。
- **管线工程化**：把 ingestion 做成可重跑的脚本/任务，支持**增量**（只处理新增/变更文件）和**删除**（源文档删了，向量也要删）。
- **元数据**：每个 chunk 带上来源/章节/时间/权限，检索时可**按元数据过滤**（见优化）。

```js
// ingestion 管线的骨架（与具体存储解耦，便于将来替换）
async function ingest(filePath, store) {
  const raw = await load(filePath)              // 读文件
  const text = parse(raw, filePath)             // PDF/HTML/MD → 纯文本
  const chunks = chunk(text, { size: 800, overlap: 100, by: 'heading' })
  for (const c of chunks) {
    const vector = await embed(c.text)
    await store.upsert({                        // upsert 而非 insert，支持增量更新
      id: c.id, text: c.text, vector,
      meta: { source: filePath, title: c.title, updatedAt: c.updatedAt },
    })
  }
}
```

## 4. 优化（决定知识库好不好用的关键，按性价比排序）

### 4.1 切分（chunking）—— 回报最高
- **大小与重叠**：太大→检索粒度粗、噪声多；太小→语义断裂。常见 300–1000 token，配 10–20% **重叠**防止句子被切断。
- **按结构切**：优先按标题/段落/语义切，而不是死按字数切。
- **小块检索、大块喂**（small-to-big / parent-child）：用小块做精准命中，但喂给模型时带上它所属的**父块/上下文**，兼顾召回精度和上下文完整。

### 4.2 检索质量
- **混合检索 (hybrid)**：向量（语义） + BM25（关键词）。向量擅长"意思相近"，关键词擅长"专有名词/精确匹配"，合起来召回更全。
- **元数据过滤**：先按 `source/time/权限` 等硬条件筛，再在子集里做向量检索（又快又准，还能做权限隔离）。
- **Top-K 调优**：K 太小漏，太大引入噪声+占上下文。常配合下面的 rerank。

### 4.3 重排序（rerank）
检索回来的候选（如 Top-20）再用一个 **rerank 模型/交叉编码器**精排，取前几条。显著提升"最相关的真排在前面"。

### 4.4 查询侧改写
- **Query rewriting / 扩展**：把口语化问题改写成更适合检索的形式；或生成多个变体查询（multi-query）合并结果。
- **HyDE**：先让 LLM "假想一个答案"，用这个假答案去检索（常比用原始问题召回更准）。

### 4.5 层级 / 摘要索引
为长文档先建"章节摘要层"，检索时**先命中摘要、再下钻到正文**（即知识点 06 里 H-MEM 的四级思路）。提升大库的检索效率与精度。

### 4.6 可溯源（citation）
每个 chunk 带来源，回答时让模型**标注引用**（知识点 07 已示范）。这是知识库类应用的刚需——用户要能核对。

### 4.7 新鲜度与维护
- **增量更新 + 失效**：源变了要 re-embed；源删了要删向量；用 `updatedAt` 标记时效。
- **去重**：同一内容多份会污染 Top-K。

### 4.8 评估（别凭感觉调）
- **检索层**：用一组"问题→应命中文档"的标注集，量 recall / precision（命中率）。
- **端到端**：答案是否忠于检索内容（faithfulness）、是否答到点（answer relevance）。
- 没有评估，上面所有优化都是盲调。

### 4.9 缓存
高频问题的 embedding / 检索结果 / 答案做缓存，省延迟和成本。
